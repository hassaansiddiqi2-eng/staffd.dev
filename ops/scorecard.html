01
CV + GITHUB AUDIT
ASYNC · 30 MIN REVIEW · PASS THRESHOLD: 16/25
—
▼
CODE QUALITY SIGNALS
Real shipped code in production
Evidence of deployed apps, live URLs, real user-facing features. Not tutorials, not forks, not "personal projects" with 1 commit. You need to find something users touched.
GitHub commit history depth
Look at commit messages — are they meaningful ("fix auth middleware token expiry bug") or lazy ("update", "fix", "changes")? Commit message quality predicts async communication quality.
Code organisation and README quality
Does their repo have a README? Is it useful? Is the folder structure logical? Engineers who document for others are engineers who communicate with clients.
EXPERIENCE SIGNALS
Employment timeline consistency
Are there unexplained gaps? Do the dates on LinkedIn match the CV? Short stints (under 6 months) at multiple employers is a yellow flag — ask about each one.
Stack depth vs stated level
Does their GitHub actually show 4+ years of React experience if they claim senior? Or is it Surface-level usage of frameworks? Self-reported seniority is often inflated by 1 level.
Pass threshold: 16 / 25 — candidates scoring below 16 are rejected at Stage 1. No exceptions.
SCORES: 1=Absent · 2=Weak · 3=Average · 4=Strong · 5=Exceptional

02
COMMUNICATION INTERVIEW
LIVE CALL · 45 MIN · PASS THRESHOLD: 20/30
—
▼
VERBAL ENGLISH
Spoken English fluency
Can you understand every sentence without effort? 5 = native-level or near. 4 = occasional accent, fully understandable. 3 = understandable but requires focus. 2 = frequently unclear. 1 = hard to follow. US clients will be on video calls with this person weekly.
Technical explanation clarity
Ask: "Walk me through a technical decision you made recently." Can they explain the tradeoff clearly without jargon overload? Do they check for understanding? This predicts how well they'll communicate blockers to a non-technical founder.
ASYNC & WRITTEN COMMUNICATION
Async writing test — Slack update quality
Send this scenario via email before the call: "You're 2 days into a feature. You've hit a blocker — the third-party payment API is returning a 403 you can't explain. Write the Slack message you'd send your client."
5 = specific, explains what was tried, gives a plan and ETA. 1 = "blocked, will update later."
Proactive communication instinct
Ask: "Tell me about a time something went wrong on a project. When did you tell your manager/client?" Proactive communicators tell before they're asked. Reactive communicators wait until it's unavoidable. You want the former.
PROFESSIONAL MATURITY
Handling disagreement professionally
Ask: "A founder tells you to build X. You think it's the wrong approach. What do you do?" Red flag: "I just build what I'm told." Red flag: "I'd argue until they agreed." Green flag: structured pushback with alternatives, then deference to their decision.
Self-awareness about skill gaps
Ask: "What's a technology in your stack you feel weakest in right now?" Engineers who can't answer this are either dishonest or lack self-awareness — both are dangerous. Engineers with honest, specific answers are safe to match.
Pass threshold: 20 / 30. Any individual criterion below 3 triggers a review conversation — even if total score passes.

03
TECHNICAL ASSESSMENT
ASYNC · 48HR TASK · SCORED BY TECHNICAL REVIEWER · PASS THRESHOLD: 28/40
—
▼
PROBLEM SOLVING
Correct solution to primary task
Did they solve the actual problem set? Reviewer grades: 5 = complete + elegant. 4 = complete, some inefficiency. 3 = mostly correct, minor gaps. 2 = partially correct. 1 = incorrect or incomplete.
Edge cases and error handling
Did they handle null values, empty states, API failures? Senior engineers think about what breaks. Juniors think about the happy path. This criterion separates levels more than anything else.
Written reasoning quality
Did they explain their approach? We ask candidates to write a brief explanation of their solution. This doubles as a communication test. The explanation matters as much as the code.
CODE QUALITY
Code readability and naming
Would another developer understand this code without asking questions? Variable names, function names, structure. Code that reads like English is code that scales in a team environment.
Appropriate use of patterns
Did they use React hooks correctly? Avoid unnecessary re-renders? Separate concerns properly? Reviewer checks for anti-patterns: prop drilling, useEffect abuse, massive components, hardcoded values.
Performance awareness
Did they think about performance? Memo, callbacks, avoiding unnecessary renders? Not required at Junior level — but at Mid and Senior it's expected. Reviewer adjusts this score against the stated level.
Testing consideration
Did they write tests, or at minimum acknowledge they would? Even a comment like "// I'd add Jest unit tests here for X" shows the right instinct. Zero mention of testing is a yellow flag at Senior level.
REVIEWER CONFIDENCE
Technical reviewer overall confidence
Reviewer's gut: "Would I want this person in my team?" 5 = Yes, immediately. 4 = Yes with minor reservations. 3 = Neutral. 2 = Hesitant. 1 = No. This is the reviewer's subjective summary — weight it heavily.
Pass threshold: 28 / 40. Reviewer fills this scorecard. COO reviews and confirms. Technical reviewer fee: PKR 4,000 on confirmation of pass/fail.

LIVE CODING INTERVIEW

—
▼
LIVE BUILD QUALITY
Initial approach and planning
Before writing a line of code, did they ask clarifying questions? Did they outline their approach? Engineers who plan before typing produce better code and have better client relationships.
Code quality under time pressure
Live conditions reveal real habits. Do they revert to messy code when stressed, or maintain standards? The way they code in an interview is the floor — real work will be at or above this level.
Task completion within 60 minutes
Did they get to a working solution? Partial completion is not a failure if their process was strong and they communicated clearly about trade-offs. A complete but poorly explained solution scores lower than a partial but well-narrated one.
THINKING PROCESS
Narration when stuck
When they hit a wall, do they go silent or narrate their thinking? "I'm going to try X because Y... actually that won't work because Z, let me try A instead" is exactly what a remote team needs to see. Silence is disqualifying.
Response to mid-session correction
Interviewer introduces a deliberate constraint halfway through ("actually the client wants this to be reusable across 3 components"). Good engineers adapt without frustration. Great engineers say "good catch" and pivot cleanly.
Post-build reflection quality
At the end: "If you had another hour, what would you improve?" Engineers who can self-critique accurately are engineers who grow. Engineers who say "nothing, it's fine" are engineers who plateau.
Pass threshold: 20 / 30. Any candidate scoring below 3 on "Narration when stuck" is automatically rejected regardless of total score.
